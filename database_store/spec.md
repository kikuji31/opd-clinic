# Spec: ระบบฐานข้อมูลจัดเก็บข้อมูลคนไข้คลินิกเสริมความงาม

## 1. ภาพรวมโครงการ

ออกแบบฐานข้อมูลพื้นฐาน (basic) สำหรับคลินิกเสริมความงาม เพื่อจัดเก็บข้อมูลคนไข้, ประวัติการรักษา/ทำหัตถการ,
การนัดหมาย, รายการบริการ, พนักงาน/แพทย์ผู้ให้บริการ และการชำระเงิน

**ขอบเขต:** เป็นระบบพื้นฐานสำหรับจัดเก็บและเรียกดูข้อมูลเท่านั้น **ยังไม่รวมการกำหนดสิทธิ์การเข้าใช้งาน (role-based
access control)** — ทุกผู้ใช้งานระบบถือว่าเข้าถึงข้อมูลได้เท่ากันในเวอร์ชันนี้ สามารถเพิ่ม layer สิทธิ์การเข้าใช้งาน
ในเวอร์ชันถัดไปได้

**ฐานข้อมูลเป้าหมาย:** PostgreSQL บน Neon (serverless Postgres)

## 2. ความต้องการพื้นฐานของระบบ (Requirements)

1. บันทึกและค้นหาข้อมูลคนไข้ได้ (ข้อมูลส่วนตัว, ประวัติแพ้ยา/สารเคมี, โรคประจำตัว, ผู้ติดต่อฉุกเฉิน)
2. จัดการข้อมูลพนักงาน/แพทย์ผู้ให้บริการ
3. กำหนดรายการบริการ/หัตถการที่คลินิกให้บริการ พร้อมราคาและระยะเวลาโดยประมาณ แยกตามหมวดหมู่
4. บันทึกการนัดหมายล่วงหน้าของคนไข้ พร้อมสถานะการนัด (นัดแล้ว/ยืนยันแล้ว/มาแล้ว/ยกเลิก/ไม่มา)
5. บันทึกประวัติการรักษา/ทำหัตถการจริงของคนไข้แต่ละครั้ง (รายละเอียด, ผลลัพธ์, ผลข้างเคียงถ้ามี,
   คำแนะนำนัดครั้งถัดไป) เพื่อใช้อ้างอิงประวัติการรักษาในอนาคต
6. บันทึกการชำระเงินที่ผูกกับการรักษาแต่ละครั้ง (จำนวนเงิน, วิธีชำระ, สถานะการชำระ)
7. รองรับกรณีคนไข้เดิน walk-in (ทำหัตถการโดยไม่ผ่านการนัดหมายล่วงหน้า)

**นอกขอบเขตของเวอร์ชันนี้:** ระบบสิทธิ์ผู้ใช้งาน (login/role), ระบบสต็อกสินค้า/เวชภัณฑ์, ระบบแจ้งเตือน,
multi-branch (หลายสาขา)

## 3. โครงสร้างฐานข้อมูล (Schema Design)

### ตาราง `patients` — ข้อมูลคนไข้
| คอลัมน์ | ชนิดข้อมูล | คำอธิบาย |
|---|---|---|
| patient_id | SERIAL PK | รหัสคนไข้ |
| hn_code | VARCHAR(20) UNIQUE | เลขประจำตัวคนไข้ (HN) |
| first_name / last_name | VARCHAR(100) | ชื่อ-นามสกุล |
| nickname | VARCHAR(50) | ชื่อเล่น |
| gender | VARCHAR(10) | เพศ |
| date_of_birth | DATE | วันเกิด |
| phone / email / address | - | ข้อมูลติดต่อ |
| id_card_number | VARCHAR(20) | เลขบัตรประชาชน |
| allergy_history | TEXT | ประวัติแพ้ยา/สารเคมี |
| underlying_disease | TEXT | โรคประจำตัว |
| emergency_contact_name / phone | - | ผู้ติดต่อฉุกเฉิน |
| registered_date | DATE | วันที่ลงทะเบียน |
| notes | TEXT | หมายเหตุอื่นๆ |

### ตาราง `staff` — พนักงาน/ผู้ให้บริการ
แพทย์, พยาบาล, นักบำบัดผิว — เก็บชื่อ, ตำแหน่ง, เลขใบประกอบวิชาชีพ (ถ้ามี), เบอร์โทร, สถานะใช้งาน

### ตาราง `service_categories` — หมวดหมู่บริการ
เช่น โบท็อกซ์, ฟิลเลอร์, เลเซอร์, ดูแลผิวหน้า

### ตาราง `services` — รายการบริการ/หัตถการ
ผูกกับหมวดหมู่ (category_id) — เก็บชื่อบริการ, รายละเอียด, ราคา, ระยะเวลาโดยประมาณ (นาที), สถานะเปิด/ปิดให้บริการ

### ตาราง `appointments` — การนัดหมาย
ผูกคนไข้ (patient_id), บริการ (service_id), ผู้ให้บริการ (staff_id) เข้าด้วยกัน พร้อมวันเวลานัด และสถานะ

### ตาราง `treatment_records` — ประวัติการรักษา/ทำหัตถการจริง
ผูกกับ appointment (ถ้ามี — เผื่อกรณี walk-in ไม่ต้องมี), คนไข้, บริการ, ผู้ให้บริการ พร้อมรายละเอียดการทำหัตถการ,
ผลลัพธ์, ผลข้างเคียง (ถ้ามี), และวันที่แนะนำให้มาติดตามผลครั้งถัดไป

### ตาราง `payments` — การชำระเงิน
ผูกกับ treatment_record หนึ่งรายการ เก็บจำนวนเงิน, วิธีชำระ (เงินสด/บัตรเครดิต/โอนเงิน/พร้อมเพย์), สถานะการชำระ

### ความสัมพันธ์ระหว่างตาราง (Relationship)

```
patients ──< appointments >── staff
   │             │
   │             └──< treatment_records >── staff
   │                        │
   └────────────────────────┘
                             │
services ──< appointments, treatment_records
service_categories ──< services

treatment_records ──< payments
```

## 4. การนำไปใช้งานจริง

- Schema และข้อมูลตัวอย่างถูกสร้างไว้บน Neon PostgreSQL แล้ว ที่ฐานข้อมูล `neondb`
- ไฟล์ SQL ต้นฉบับ (DDL + sample data): ดูรายละเอียดโครงสร้างตารางแบบเต็มได้จาก schema ที่ deploy จริงในฐานข้อมูล
- ข้อมูลตัวอย่างที่ใส่ไว้:
  - พนักงาน 3 คน (แพทย์ 2, พยาบาล 1)
  - หมวดหมู่บริการ 4 หมวด, บริการ 8 รายการ
  - คนไข้ตัวอย่าง 5 คน
  - การนัดหมาย 6 รายการ
  - ประวัติการรักษาจริง 3 รายการ (พร้อมการชำระเงินที่ผูกกัน)

## 5. แนวทางขยายในอนาคต (ไม่อยู่ในขอบเขตเวอร์ชันนี้)

- เพิ่มระบบ user/role สำหรับกำหนดสิทธิ์การเข้าถึงข้อมูล (เช่น แอดมิน, แพทย์, พนักงานต้อนรับ)
- เพิ่มระบบสต็อกเวชภัณฑ์/ผลิตภัณฑ์ที่ใช้ในการรักษา
- เพิ่มระบบแจ้งเตือนนัดหมายผ่าน SMS/LINE
- รองรับหลายสาขา (multi-branch)
- เพิ่มระบบแนบไฟล์รูปภาพก่อน-หลังการรักษา
